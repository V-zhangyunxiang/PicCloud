# 什么是 FCM
# FCM Push 依赖 App 存活吗

## 核心结论

国外的 Android App 通过 FCM 发送推送通知，完全可以不依赖 App 自身进程存活，并支持在后台配置标题、文案、图片、跳转链接等，这与国内厂商通道的能力是对等的。

FCM 与国内厂商通道的技术原理是一致的。在支持 Google Play 服务（GMS）的设备上，`Google Play 服务` 作为系统级进程负责与 FCM 服务器维持长连接。因此，即便你的 App 进程因各种原因被结束，FCM 通知也能正常下发。这个核心结论有明确的官方依据：FCM 的“通知消息”由系统直接接管和展示，其整个收发过程不依赖于你的应用是否存活。

## 内容配置

- **标题、文案、图片**：这些内容都可以通过 FCM 后台发送的 **"通知消息"** 的 `notification` 字段进行配置。同时，FCM 也支持 `BigPictureStyle` 等样式来展示丰富的内容。国内厂商对推送图片通常有大小、数量、分辨率等限制，而 FCM 的通知负载有 4KB 的上限。

- **跳转链接与自定义行为**：FCM 在配置跳转上比国内厂商更灵活，支持两种模式:
    1. **直接链接跳转**：在 `notification` 负载中直接设置 `click_action` 等字段，让系统直接打开指定链接或 App 特定页面。
        
    2. **数据消息处理**：发送 `data` 消息，由你的 App 在 `onMessageReceived` 回调中解析数据，构建 `PendingIntent` 并展示。不过，**`data` 消息的处理依赖于 App 进程存活。**

## 两种消息类型

FCM 提供“通知消息”和“数据消息”两种类型以满足不同需求。

- **通知消息**：实现你目标的标准方案。消息包含标题、内容、图片等，即使 App 被杀死，系统也能直接接管展示。
    
- **数据消息**(透传消息)：用于需要自定义处理逻辑的场景，由 `data` 字段承载自定义键值对。收到后需在你的 App 代码中处理才能展示通知，**App 被杀时完全无法接收和处理**。
    
- **混合消息**：包含 `notification` 和 `data` 负载，可以兼顾系统展示和自定义数据处理。

## 总结

FCM 在实现“不依赖 App 存活的后台配置通知”这个核心需求上，能力与国内厂商通道是一致的，它的实际可行性高度依赖于设备环境——**必须完整内置 Google Play 服务**。而国内 Android 设备通常阉割或禁用 GMS，这正是国内厂商通道产生的根本原因。

如果你的 App 面向海外市场，FCM 会是标准且强大的解决方案；如果面向国内，选择集成各大手机厂商的 Push SDK 会是更稳妥的做法。

# 通知消息

**“通知消息”是所有厂商都支持的通用语言**，用于保证消息的可靠送达；而**透传消息则可以满足更灵活的开发需求**。**“通知消息”的处理往往被下放到系统框架层**，与应用的直接交互（回调）是可选的，并且各家有不同的实现逻辑。所以，在实际开发时，需要仔细查阅并遵循各家厂商的文档。

**点击处理**

通知消息的点击处理，主流做法是依赖 **Intent** 机制，各厂商的实现路径虽有不同，但最终都会回到 `Intent` 这个核心上。

无论哪个厂商，在用户点击通知后，系统最终都会启动你应用的某个 `Activity`，并把数据封装在 `Intent` 里。因此，开发者的核心任务就两个：**① 在 AndroidManifest.xml 中为 Activity 配置好 intent-filter；② 在目标 Activity 的 `onCreate` 或 `onNewIntent` 方法中解析 Intent。** 这是最底层、最通用的兜底方案。

# 透传消息

透传消息的处理流程：

1. **数据接收**：App 在 `onMessageReceived` 或其对应的回调方法中获取推送数据。
    
2. **逻辑处理**：解析数据，执行静默操作（例如：记录日志、从服务器拉取新数据）。
    
3. **创建通知**：利用 Android 的 `NotificationManager` 和 `Notification.Builder` API，自行构建并发送一个系统通知。

- **透传消息**：通过 `onMessageReceived` 回调后，App 可以**完全自定义**通知的 UI（例如使用 `RemoteViews` 制作自定义布局、添加多个按钮、修改字体颜色等），也可以展示 App 内的非系统级横幅（如 `Toast`、`Snackbar` 或自定义浮层）。但前提是 **App 进程必须存活**，否则回调无法触发。
    
- **通知消息**：由系统服务直接展示，使用**系统默认的样式**（标题、内容、小图标等），虽然可以设置大图、分组等，但无法做到透传消息那样的完全自定义 UI。它的最大优势是 **App 无需存活**，即使进程被杀死也能正常展示。

- **需要完全自定 UI / 实时逻辑处理** → 使用透传消息，并做好进程保活。
    
- **只需要标准样式，且希望高送达率** → 使用通知消息，享受系统级通道的稳定性。

# 如果 App 有自己的长连接逻辑，还需要 FCM 吗

**FCM 最大的价值，就是解决了你自己写长连接时，最头疼的两个“不可能”问题：App 被杀死后如何收到消息，以及如何绕开手机厂商严苛的后台省电限制。**

要理解为什么，我们先得明白自建长连接的致命缺陷。

- **App 被杀死后，一切归零**：自建长连接依赖于你的 App 进程，进程被用户或系统杀死，连接就断了，推送通道随之消失，所有相关的逻辑都会失效。而 FCM 依赖的是系统级进程（Google Play 服务），不受此影响。
    
- **无法绕开的“电量与后台限制”**：在国产手机上，为了让 App 后台长时间联网，开发者需要与厂商的省电策略“斗智斗勇”，维护成本极高。FCM 作为系统级服务，不受这些严格限制。

这可以理解为：**自建长连接只是你和服务器之间的一条“专线”，而FCM是手机系统内部的一条“免费VIP通道”**。走自己的“专线”需要自己打点沿途关系和支付高昂的成本（如电量和保活），而走“VIP通道”则无需关心这些，系统会帮你搞定一切。

**什么情况下需要自建长连接**

虽然 FCM 是主力，但在以下场景，自建长连接依然是必要的：

- **高并发实时通信**：对于即时通讯（IM）等对实时性和数据交互有极高要求的场景。
    
- **自有协议或加密**：当业务对数据传输的协议和加密有严苛要求，FCM 标准通道无法满足时。
    
- **Google服务不可用地区**：主要在**国内**，由于网络原因 FCM 无法稳定使用，必须适配国内厂商通道或自建长连接。有趣的是，微信这类体量的App，即便在国内也采用了“**自建长连接为主 + 系统推送为辅**”的混合方案，但它们的自建长连接由极高投入的顶级技术团队维护，应用常驻后台并有庞大的装机量保证其进程优先级，其策略对绝大多数App而言难以复制。


# 什么是长连接？与短连接的区别是什么？为什么需要长连接？

- **长连接（持久连接）**：通信双方（如 App 与服务器）建立连接后，**不主动断开**，双方可以随时向对方发送数据。它通常基于 TCP 协议实现，也有基于此的 WebSocket、MQTT 等。为了维持这条“活路”，客户端需要定时发送**心跳包**，告诉服务器“我还活着，别断开我”。
    
- **短连接（一次性连接）**：每次数据交互都是一次独立的请求。App 发起请求 -> 服务器响应 -> 连接立即关闭，各不相干。我们平时调用的大部分 HTTP RESTful API（获取天气、登录、提交表单），走的就是短连接。

**核心区别对比**

| 对比维度           | 📡 长连接 (长链接)                               | ⚡ 短连接 (短链接)                           |
| -------------- | ------------------------------------------ | ------------------------------------- |
| **生命周期**       | 建立后长久保持，直到主动断开或网络异常                        | 请求发起时建立，响应返回后立即关闭                     |
| **通信模式**       | **双向通信**。服务器可以随时“主动”推数据给客户端                | **单向请求-响应**。必须由客户端发起，服务器被动应答          |
| **实时性**        | **高**。消息到达服务器端即可立刻下发，延迟在毫秒级                | **低**。如果想获取新消息，客户端必须不断轮询（拉取），浪费资源且延迟高 |
| **资源消耗（客户端）**  | 占用内存和系统线程，且需定期发心跳包（耗电、耗流量）                 | 资源占用极低，用完即释放                          |
| **资源消耗（服务器端）** | 占用大量文件描述符和内存，需要维护海量连接状态                    | 资源占用极低，无状态，便于横向扩容                     |
| **抗网络波动性**     | **弱**。移动网络切换（WiFi<->4G）、信号弱时极易断开，需要复杂的重连机制 | **强**。每次独立请求，不受前一次网络状态影响              |
| **典型应用场景**     | 即时通讯(IM)、实时定位、股票行情、**推送服务**                | 普通 API 数据获取、网页浏览、文件下载                 |

# 在 Android 中如何自建长连接，有哪些常见和主流的实现方式

在 Android 中自建长连接，主流实现方式可以分为四个梯队，按从“简单通用”到“复杂专业”的顺序分别是：**WebSocket、MQTT、原生 TCP Socket 和 Netty**。

这四种方式各有优劣，适用于不同的业务场景。

### 🏆 主流实现方式对比

| 实现方式 | 核心特点 | 优势 | 劣势 | 适用场景 |
| :--- | :--- | :--- | :--- | :--- |
| **WebSocket** | 基于 TCP 的全双工协议，通过 HTTP 升级握手建立连接。 | **开发简单**，OkHttp 等主流库原生支持；**兼容性好**，使用 80/443 端口，不易被防火墙屏蔽。 | 需处理协议升级；报文头部相对较重，有额外开销。 | **即时通讯 (IM)**、**实时推送**、**股票行情**、**在线游戏**等需要高频双向通信的场景。 |
| **MQTT** | 轻量级的**发布/订阅**消息协议，专为低带宽、高延迟、不稳定的网络环境设计。 | **极低功耗**，报文头最小仅 2 字节；**弱网友好**，支持三种 QoS 级别，确保消息可靠送达；**内置心跳和遗嘱**机制。 | **需要部署 Broker 服务器**；协议本身有一定复杂度。 | **物联网 (IoT)**、**智能硬件**、**车联网**、**传感器数据上报**等对功耗和网络要求苛刻的场景。 |
| **原生 TCP Socket** | 直接使用 `java.net.Socket` 和 `ServerSocket` 进行编程。 | **最高灵活性**，可**完全自定义**二进制协议；**性能最优**，无任何额外封装开销。 | **开发成本极高**，需自行处理粘包/拆包、心跳、断线重连等所有底层细节。 | **自定义二进制协议**、**金融交易系统**等对协议和性能有极端要求的场景。 |
| **Netty** | 基于 Java NIO 的**高性能、异步事件驱动**的网络框架。 | **高性能、高并发**；**协议高度可定制**，支持 TCP/UDP/HTTP/WebSocket 等；社区成熟，有 ANetty 等 Android 封装库。 | **学习曲线陡峭**，需深入理解 NIO 和事件驱动模型；**增加 APK 体积**（约 4MB+）。 | **即时通讯服务器**、**游戏服务端**、**物联网网关**等需要处理海量连接的高性能场景。 |

### 🛠️ 核心实现步骤（以最通用的 WebSocket 为例）

无论选择哪种方式，实现一个稳定的长连接都需要完成以下核心步骤。以 Android 上最常用的 **OkHttp + WebSocket** 为例：

1.  **引入依赖**：在 `build.gradle` 中添加 OkHttp 依赖。
2.  **建立连接**：使用 `OkHttpClient` 的 `newWebSocket()` 方法创建连接，并传入 `WebSocketListener` 监听连接状态和消息。
3.  **实现心跳**：**这是保证连接存活的关键**。OkHttp 提供了 `pingInterval` 方法设置自动心跳间隔，这是最简单的保活方式。
4.  **实现重连**：在 `onClosed()` 或 `onFailure()` 回调中，使用**指数退避算法**等策略尝试重新建立连接。
5.  **管理生命周期**：在 Activity 或 Service 的 `onDestroy()` 等方法中主动关闭连接，避免资源泄漏。

### 💎 总结与选型建议

选择哪种方式，本质上取决于你的**核心需求**和**团队能力**：

*   **追求开发效率和通用性**：**WebSocket** 是最佳选择。它简单、现代，能满足大部分 App 的实时通信需求。
*   **面向 IoT 或弱网环境**：**MQTT** 是专业之选，它的低功耗和高可靠性是其他方案无法替代的。
*   **需要极致的性能和完全的自定义协议**：可以考虑 **原生 TCP Socket** 或 **Netty**。但这要求团队有深厚的网络编程功底，并准备好应对高昂的开发和维护成本。

另外提醒一下，自建长连接在 Android 系统（尤其是国内定制系统）上，最大的挑战往往是**应用后台存活**和**省电策略**。你需要配合使用**前台服务 (Foreground Service)** 等方式，并针对不同厂商做适配，才能保证长连接的稳定性。这也是我们之前讨论中，FCM 等系统级通道能体现其核心价值的地方。
