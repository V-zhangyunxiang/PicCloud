
# 启动时间测量方式

1. **ADB命令方式**：
    - 通过`adb shell am start -W`命令，可以在终端中启动应用程序并获取到启动时间。
    - 输出的时间值包括`ThisTime`(最后一个 Activity 耗时)、`TotalTime`(所有 Activity 启动耗时)和 `WaitTime`(AMS 启动 Activity 的总耗时)。在没有 Splash 界面的应用中，`thisTime` 与`totalTime`是一致的；`awaitTime` 则通常更长一些。
    - ADB 命令适合线下测试，但不适合线上使用，因为它不能在实际用户设备上执行，并且提供的启动时间数据不够精确。

2. **手动打点方式**：
    - 在代码中记录开始时间和结束时间以计算启动耗时。
    - 启动开始时间建议在`attachBaseContext`方法里记录，而结束时间应当选择在内容首次展示给用户的时间点(例如 Feed 的第一条显示)，而不是首帧绘制完成时(如`onWindowFocusChanged`事件)。
        在 list position =0 时，使用 viewTree 的 getViewTreeObserver 添加 addOnPreDrawListener 监听，回调中打结束点(结束后需要 remove Listener)。
    - 手动打点的方式更加准确，可以用于线上环境，并能通过服务器收集和分析这些数据来进一步优化用户体验。

# 启动优化工具选择

1. **traceview**

   图形的形式展示执行时间，调用栈，包含所有线程的信息。
   运行时开销严重，整体会变慢，可能会带偏优化方向。

   **使用方式**
    Debug.startMethodTracing("") //文件名
    Debug.stopMethodTracing()    //生成文件
    生成文件在 sd卡：Android/data/packagename/files/文件名

   **文件使用**

   通过 Device File Explorer 访问，提供了 **Call Chart**、Flame Chart、**Top Down** 及 Bottom Up 四种视图帮助分析函数调用情况及其时间消耗。

   **几个时间的区别**

  - **Self Time vs. Total Time**：Self Time 指函数自身执行时间；Total Time 则包括了函数内部调用其他函数的时间。
  - **Wall Clock Time vs. Thread Time**：前者代表线程实际运行时间，后者指 CPU 真正花费的时间。

2. **systrace**

   结合 Android 内核的数据，生成 HTML 报告。
   API 18 以上使用，推荐 TraceCompat。
   轻量级、开销小。
   直接显示 CPU 利用率。

   **使用方式**

   - python systrace.py。 -b 表示 buffer 收集多少，-t 表示时间，-a 表示检测的包名，-o 表示 output 生成的文件。
   - TraceCompat.benginSection("ApponCreate")   //起点位置
   - TraceCompat.endSection();
   - 使用浏览器打开 HTML 文件。

   **WallTime 和 CPU Time 区别**

   WallTime 是代码执行时间。CPUTime 是代码消耗 CPU 的时间(重点指标)。

# 获取方法耗时

 1. 方法手动打点
 
- 每个方法执行前后打前后时间戳，计算耗时。
- 侵入性强，工作量大。

2. AOP 切面编程

   针对同一类问题的统一处理，无侵入代码。

   **AspectJ  使用**

   实现 AOP 的组件库。

  - join points
    可以作为切面的地方。函数的调用、执行，set/get 变量，类初始化。
 - pointCut
   带条件的 join points。
 - advice
   一种 Hook，要插入代码的位置。
   Before：在 pointCut 之前执行。
   After: 在 pointCut 之后执行。
   Around: 在 pointCut 前后分别执行。

```java
@Around("call (@com.example.annotation.TimeTrace**(..))")
public Object aroundMethod(ProceedingJoinPoint joinPoint) throws Throwable {
        
        // ① 方法执行【前】的逻辑
        long startTime = System.currentTimeMillis();
        String methodName = joinPoint.getSignature().getName();
        
        // ② 执行原方法（必须调用，否则原方法不会执行）
        Object result = joinPoint.proceed();
        
        // ③ 方法执行【后】的逻辑
        long cost = System.currentTimeMillis() - startTime;
        Log.d("TimeTrace", methodName + " 耗时: " + cost + "ms");
        
        return result; // 返回原方法结果
}
```

# 常规异步 + 启动器

常规异步通过开启多个线程池任务，分别执行各自的逻辑。

**常规异步痛点**

- 代码不优雅。
- 场景不好处理(依赖关系)
- 维护成本高。

**启动器作用**

充分利用 CPU 多核自动梳理任务顺序。

**实现流程**

1. 代码 Task 化，启动逻辑抽象为 Task。

2. 根据所有任务依赖关系排序生成一个有向无环图。

3. 多线程按照排序后的优先级依次执行。

**代码逻辑**

- 使用入口类 `TaskDispatch` 来负责启动器的初始化和调度工作。
- 通过 `addTask` 方法将需要执行的任务添加到启动器中。
- 应用 TOP 排序算法来确定任务的执行顺序。
- 调用 `Start` 方法开始执行已排好序的任务列表。

# 延迟初始化

在优化 Android 应用启动性能时，识别出一些低优先级任务可以被延迟执行，以提升用户体验。

**常规方案及其问题**：
- 直接将**低优先级任务移至Feed 首帧显示后**执行或使用`Handler.postDelayed()`来延迟执行这些任务。
- 这些方法可能导致主线程卡顿，影响用户界面交互，并且代码不易维护。

**更优解决方案**：
- 引入了分批初始化的概念，结合系统的 `IdleHandler` 特性，在系统空闲时逐步执行任务。

- 创建了一个 `DeleteInitStarter` 类，用于管理任务的添加与执行。通过链式调用机制简化任务添加流程，并利用 `LinkedList` 存储待处理的任务列表。

- 使用 `Looper.myQueue().addIdleHandler()` 注册 `IdleHandler`，确保只有当系统处于空闲状态时才执行任务，从而避免干扰用户的操作体验。

- **每次只从队列中取出一个任务执行**，根据队列是否为空决定是否需要移除`IdleHandler`，保证了应用程序的流畅性。

# 启动优化方案总结

三部曲：异步、延迟、懒加载。

核心思想: 子线程分担主线程任务，并行减少时间。

- 冷启阶段未使用到，且不需要主线程的任务。使用线程池异步执行。

- 冷启阶段未使用到，但必须在主线程的任务。通过 **IdleHandler 延迟到系统空闲时**在主线程执行。

- 冷启阶段使用到了，必须在主线程执行的任务。**将其中不依赖主线程的耗时部分提前在子线程预处理，主线程只执行最终必要的轻量操作**。无法拆分时，接受现状，确保该任务本身尽量精简。

- 冷启阶段使用到了，不需要主线程的任务。线程池异步执行，但需要配合等待机制确保主线程使用结果前异步任务已执行完毕。**也是启动器解决的核心问题**。

 - 启动器。将功能抽象为一个 Task，生成一个有向无环图，自动拓扑排序，自动处理依赖、并发、等待。线程池核心数根据设备 CPU 数量设置(IO 型或者 CPU 型耗时任务)，开启多个线程任务分别执行，充分利用线程并发能力。

- 在子线程提前触发 SP 加载，主线程读 SP 直接从内存返回。

- 启动阶段不启动子进程。子进程会共享 CPU 资源，导致主进程 CPU 紧张。
